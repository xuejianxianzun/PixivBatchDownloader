# 测试 worker 里的 setTimeout 定时器的表现

当标签页处于后台时，windows 上的 setTimeout 和 setInterval 的回调函数会被大幅延迟。这个问题我在 3 年前已经测试过:[页面处于后台时，定时器的延迟问题.md](./页面处于后台时，定时器的延迟问题.md)

当时的解决方法是使用 worker 里的定时器替代 windows 上的定时器，有一定作用，因为当标签页处于后台时，worker 里的定时器不容易受到影响。

现在 **2026-04-19** 我又进行了测试，验证此方法是否依然有效。

## 测试 1600 ms 间隔

流程是在一个用户主页抓取 2 页共 96 个作品，慢速抓取，间隔时间是默认的 1600 ms。重复测试时先刷新页面。

在 getWorksData 时开始计时，crawlFinished 时输出计时结果。4 分钟后回到标签页（此时已经抓取完毕），查看控制台里输出的时间。

分别测试 3 次，表现如下（用时里包括网络请求耗费的时间）：

| worker.setTimeout 用时 | window.setTimeout 用时 |
| ---------------------- | ---------------------- |
| 179682                 | 196468                 |
| 169581                 | 193247                 |
| 172954                 | 193114                 |

worker 明显更快，平均每次请求 1800 ms，应该是 1600 ms 间隔 + 200 ms 网络请求。 window.setTimeout 的平均 2000 ms，说明其间隔时间基本就是 1800 ms。

这也符合我在 3 年前测试时的表现：当标签页处于后台时，window.setTimeout 的 1600 ms 间隔会被浏览器增加到 1800 ms。

## 测试 1000 ms 间隔


在上面的对比里，worker 的时间比 window.setTimeout 少的有限，这是因为两者在实际运行时的间隔时间相差不大（1600 vs 1800）。如果把间隔时间设置为 1000 ms，那么两者之间就会相差 800 ms（1000 vs 1800），时间差别应该很明显。

分别测试 1 次：

| worker.setTimeout 用时 | window.setTimeout 用时 |
| ---------------------- | ---------------------- |
| 116843                 | 190135                 |

符合预期：worker 明显更快，平均每次请求 1200 ms，是 1000 ms 间隔 + 200 ms 网络请求。而 window.setTimeout 的用时没有明显变化，每次的实际间隔时间还是 1800 ms。

## 结论

目前在 Chrome 浏览器上，worker 里的定时器依然不会被浏览器加大延迟（至少在下载器的使用场景里如此）。用它来替代 window.setTimeout 是有用的。
